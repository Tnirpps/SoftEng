# Проектирование производительности для ЛР5

## Цель

На этом этапе задача состоит из двух частей:

- защитить write/auth endpoints от всплесков запросов через встроенный rate limiting;
- начать разгружать PostgreSQL через первый Redis-кеш в `directory_service`.

## Hot paths

В текущей архитектуре MaDisk чаще всего вызываются:

- `POST /v1/users` - регистрация пользователя;
- `POST /v1/users/login` - получение JWT токена;
- `POST /v1/directories` - создание директории;
- `PUT /v1/directories/{directory_id}` - переименование директории;
- `DELETE /v1/directories/{directory_id}` - удаление директории;
- `POST /v1/directories/{directory_id}/move` - перемещение директории;
- `POST /v1/files` - создание файла;
- `PUT /v1/files/{file_id}` - обновление файла;
- `DELETE /v1/files/{file_id}` - удаление файла.
- `GET /v1/directories/{directory_id}/files` - получение содержимого папки.

Это именно те операции, которые:

- изменяют состояние системы;
- инициируют запись в PostgreSQL или MongoDB;
- могут использоваться злоумышленником для flood- или brute-force сценариев.

## Потенциально медленные операции

### `user_service`

- регистрация пользователя вызывает запись в PostgreSQL;
- логин выполняет поиск пользователя и проверку учетных данных;
- при массовых попытках логина сервис подвержен brute-force нагрузке.

### `directory_service`

- создание, обновление, удаление и перемещение директории используют PostgreSQL;
- получение содержимого папки также обращается к PostgreSQL за списком файлов и общим количеством элементов;
- часть операций работает в транзакциях и выполняет дополнительные проверки:
  - существование родителя;
  - проверка конфликтов имен;
  - проверка циклов при перемещении;
  - проверка дочерних элементов при удалении.

### `file_service`

- создание, обновление и удаление файлов обращаются к MongoDB;
- создание выполняет проверку конфликта имени и запись документа;
- обновление и удаление дополнительно проверяют принадлежность файла владельцу.

## Требования к производительности

В рамках лабораторной ставятся практические, а не production-grade цели:

- сервис должен оставаться отзывчивым при умеренной нагрузке;
- burst-нагрузка не должна бесконтрольно проходить в слой хранения данных;
- при превышении допустимого RPS handler должен быстро отвечать `429 Too Many Requests`;
- read-запросы не должны пострадать от ограничений на write/auth endpoints.

## Выбранная стратегия

Используется встроенный limiter `userver` через параметр `max_requests_per_second` в конфигурации handler'ов.

Преимущества такого решения:

- не требуется Redis или отдельный reverse proxy;
- не меняется прикладной код handler'ов;
- ограничение применяется адресно к конкретным endpoints;
- решение легко объяснить и продемонстрировать на защите.

Чтобы limiter не влиял на обычные интеграционные тесты, значения RPS вынесены в `config_vars`. В testsuite используются заведомо большие лимиты, а реальные значения задаются в docker-конфигах.

## Первый Redis-кеш

Для первого этапа Redis интеграции выбран только один endpoint:

- `GET /v1/directories/{directory_id}/files`

Причины выбора:

- это частый read-path;
- он напрямую нагружает PostgreSQL;
- его можно кешировать без изменения HTTP-контракта;
- он подходит для простого `Cache-Aside` без расширения архитектуры на все сервисы сразу.

### Стратегия кеширования

- стратегия: `Cache-Aside`;
- ключ: `dir-files:{owner_id}:{directory_id}:{limit}:{offset}`;
- TTL: `30s`;
- в Redis хранится полный JSON-ответ endpoint;
- пустой успешный список также кешируется.

### Почему только TTL

На этом шаге не реализуется межсервисная инвалидация, потому что изменения файлов происходят в `file_service`, а кеш находится в `directory_service`.

Поэтому выбран самый простой режим согласованности:

- данные могут быть устаревшими не дольше `30s`;
- после TTL запись автоматически протухает;
- это допустимо для первого этапа внедрения Redis, задача которого — прежде всего завести рабочую инфраструктуру кеша и снизить нагрузку на БД при повторных чтениях.

## Выбранные лимиты

### `user_service`

- `POST /v1/users` - `2 req/s`
- `POST /v1/users/login` - `5 req/s`

Обоснование:

- регистрация используется реже и является более чувствительной к спаму;
- логин должен оставаться удобным для нормального пользователя, но ограничивать burst запросов.

### `directory_service`

- `POST /v1/directories` - `10 req/s`
- `PUT /v1/directories/{directory_id}` - `10 req/s`
- `DELETE /v1/directories/{directory_id}` - `10 req/s`
- `POST /v1/directories/{directory_id}/move` - `10 req/s`

Обоснование:

- это изменяющие операции, каждая из которых обращается к PostgreSQL;
- лимит достаточно мягкий для ручного и тестового использования, но защищает от резкого всплеска.

### `file_service`

- `POST /v1/files` - `10 req/s`
- `PUT /v1/files/{file_id}` - `10 req/s`
- `DELETE /v1/files/{file_id}` - `10 req/s`

Обоснование:

- операции затрагивают MongoDB и состояние файлового каталога пользователя;
- одинаковый лимит упрощает конфигурацию и объяснение решения.

## Почему read-endpoints не ограничиваются

В этой итерации rate limiting не применяется к endpoints чтения:

- `GET /v1/users/search`
- `GET /v1/directories`
- `GET /v1/directories/{directory_id}`
- `GET /v1/directories/{directory_id}/files`
- `GET /v1/files/{file_id}`

Причины:

- цель лабораторной сейчас в первую очередь защитить state-changing операции;
- read-endpoints чаще нужны для обычной работы клиента;
- ограничения на чтение без кеширования могли бы ухудшить UX без заметной пользы для текущего этапа.

## Ожидаемый эффект

После внедрения limiter'а ожидается:

- снижение риска brute-force атак на логин;
- снижение вероятности перегрузки PostgreSQL и MongoDB при burst-запросах;
- более предсказуемое поведение сервиса под всплесками записи;
- сохранение текущего API без изменений контрактов.

После внедрения Redis-кеша для содержимого папки ожидается:

- уменьшение числа повторных обращений `directory_service` к PostgreSQL;
- более быстрые повторные запросы к одному и тому же содержимому папки;
- готовая инфраструктура Redis, которую можно дальше расширять на другие endpoints.

## Проверка результата

Эффективность реализации проверяется так:

1. Поднять сервисы в docker-стенде с боевыми значениями лимитов.
2. Отправить серию быстрых запросов в endpoints с лимитом.
3. Убедиться, что часть запросов начинает получать `429`.
4. Убедиться, что read-endpoints продолжают отвечать без новых ограничений.
5. Проверить, что обычные одиночные запросы продолжают работать как раньше.

Для Redis-кеша:

1. Выполнить `GET /v1/directories/{directory_id}/files`.
2. Проверить, что в Redis появился ключ `dir-files:*`.
3. Повторить тот же запрос и убедиться, что ответ не меняется.
4. Проверить, что другой `limit/offset` создает отдельный ключ.
5. Убедиться, что `401` и `404` не создают записей в кеше.

Для этой лабораторной этого достаточно, так как цель состоит именно в демонстрации конфигурационного rate limiting, а не в полном нагрузочном профилировании.
