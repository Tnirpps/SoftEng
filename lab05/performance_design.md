# Производительность и кэширование для ЛР5

## Зачем вообще это понадобилось

К этому моменту проект уже не выглядит как набор изолированных CRUD-ручек. У нас есть несколько сервисов, базы данных, авторизация, списки сущностей и повторяющиеся чтения. В такой схеме большая часть нагрузки обычно идет не в создание данных, а в постоянное чтение уже существующих.

Если каждый раз ходить в PostgreSQL или MongoDB за одним и тем же ответом, база быстро становится лишним узким местом. Поэтому в этой лабораторной мы сделали два практических шага:

- ограничили самые чувствительные write/auth endpoints через rate limiting;
- добавили Redis как кэш для часто читаемых данных.

## Какие места здесь самые горячие

С точки зрения обычной работы клиента чаще всего используются такие запросы:

- чтение директории по id;
- получение списка директорий;
- получение содержимого папки;
- чтение файла по id;
- логин и регистрация;
- операции создания, обновления, удаления и перемещения.

Из них особенно неприятны для хранилищ два типа запросов:

- частые чтения, которые постоянно дергают базу за одинаковыми данными;
- burst-запросы на изменение данных или авторизацию.

## Что именно мы решили кешировать

На этом этапе Redis используется не для всего подряд, а для четырех read-endpoints, где кэш действительно приносит пользу:

- `GET /v1/directories/{directory_id}`
- `GET /v1/directories?parent_id=&limit=&offset=`
- `GET /v1/directories/{directory_id}/files`
- `GET /v1/files/{file_id}`

Причина выбора простая:

- эти запросы читаются часто;
- у них хорошо определяется cache key;
- ответ можно положить в Redis как готовый JSON и не пересобирать заново;
- они не меняют состояние системы, а значит хорошо подходят для `Cache-Aside`.

## Выбранная стратегия

Для всех четырех endpoints используется `Cache-Aside`.

То есть поведение одинаковое:

1. сервис собирает ключ;
2. смотрит, есть ли готовый ответ в Redis;
3. если есть, сразу отдает его;
4. если нет, читает из базы, формирует ответ и пишет его в Redis с TTL.

Такой подход самый понятный для проекта этого масштаба. Он не требует особой магии, не меняет HTTP API и хорошо сочетается с уже существующими handler'ами.

## TTL

TTL выбраны не одинаковыми, потому что данные по природе немного разные:

- директория по id: `300` секунд;
- файл по id: `300` секунд;
- список директорий: `60` секунд;
- список файлов в папке: `30` секунд.

Логика тут такая:

- отдельная директория или файл меняются не каждую секунду, их можно держать в кэше дольше;
- списки чувствительнее к изменениям, поэтому для них TTL короче;
- список файлов в папке самый живой сценарий, поэтому у него TTL минимальный.

## Ключи кэша

Используются такие ключи:

- директория по id: `dir:get:{owner_id}:{directory_id}`
- файл по id: `file:get:{owner_id}:{file_id}`
- список директорий: `dir:list:v{version}:{owner_id}:{parent_or_root}:{limit}:{offset}`
- список файлов в папке: `dir-files:v{version}:{owner_id}:{directory_id}:{limit}:{offset}`

Отдельно для списков хранятся version-key:

- `dir:list:ver:{owner_id}:{parent_or_root}`
- `dir-files:ver:{owner_id}:{directory_id}`

Это позволяет не угадывать все старые ключи вручную. Когда список устаревает, версия увеличивается, и новые запросы автоматически начинают работать уже с новым набором ключей.

## Как устроена инвалидация

### Для директорий

- создание директории инвалидирует список у родителя;
- переименование директории инвалидирует саму директорию и список у ее родителя;
- удаление директории инвалидирует саму директорию, список у родителя и список файлов этой папки;
- перемещение директории инвалидирует саму директорию и списки у старого и нового родителя.

### Для файлов

- изменение файла инвалидирует кэш файла по id и список файлов его папки;
- удаление файла делает то же самое;
- создание файла инвалидирует список файлов папки, в которую он был добавлен.

Здесь важный момент: `directory_service` и `file_service` используют один и тот же Redis, поэтому `file_service` может сразу инвалидировать кэш списка файлов папки, который читает `directory_service`. Это избавляет нас от неприятной ситуации, когда файл уже изменился, а содержимое папки еще долго остается старым.

## Почему read-endpoints не ограничивались rate limiting

На чтение сейчас логичнее влиять кэшем, а не лимитами. Если ограничить чтение, но не ускорить его, мы просто начнем чаще отвечать отказом. Это не решает корневую проблему.

Поэтому здесь стратегия такая:

- write/auth endpoints защищаем rate limiting;
- read-endpoints разгружаем через Redis.

## Что ожидаем от такого решения

Практический эффект довольно понятный:

- повторные чтения директорий и файлов уходят из базы в Redis;
- снижается нагрузка на PostgreSQL и MongoDB;
- списки не требуют ручного удаления десятков ключей благодаря version-key;
- после изменений данных кэш очищается там, где это действительно нужно;
- архитектура остается простой: отдельные cache-компоненты, минимум логики в handler'ах.

Это не идеальная "боевая" система кэширования на все случаи жизни, но для текущего проекта это уже нормальный рабочий уровень. Дальше на этой базе можно спокойно добавлять новые cacheable endpoints, статистику hit/miss и более тонкую настройку TTL.
