# Домашнее задание 01: Документирование архитектуры в Structurizr

## Вариант 11: Система хранения файлов (Cloud Drive)

### Основная концепция системы
Система представляет собой облачное хранилище файлов (аналог Яндекс.Диска). Главная архитектурная особенность — строгое разделение логики и потоков данных. Какие функции предоставляет сервис:

* **Управление пользователями** — аутентификация и контроль выделенного места (квоты).
* **Control Plane (Метаданные)** — работа со структурой папок (создание, удаление, навигация).
* **Data Plane (Файловые данные)** — тяжёлые операции стриминга файлов, загрузка и скачивание.

**Реализованные API-сценарии:**
* **Аккаунты:** Создание пользователя, поиск по логину или ФИО.
* **Дерево каталогов:** Создание папок, удаление, просмотр содержимого папки.
* **Файловые операции:** Загрузка файла в папку, скачивание по имени, удаление файла.

### Интеграции
* **Антивирусный шлюз (ICAP/REST):** Внешнее API (например, VirusTotal или кластер ClamAV) для проверки загруженных объектов на вредоносный код.

### Контейнеры системы (уровня C2)
1. **Web UI (React/TS)** — веб-интерфейс файлового менеджера.
2. **API Gateway (Nginx)** — единый роутер. Направляет лёгкие JSON-запросы в сервисы метаданных/пользователей, а тяжёлые Multipart-запросы — в файловый узел.
3. **User Service (C++ userver)** — микросервис управления профилями и дисковыми квотами пользователей.
4. **Directory Service (C++ userver)** — управляет каталогами и связями файлов с папками.
5. **File Node (C++ userver)** — сервис, оптимизированный под I/O операции. Стримит данные в хранилище.
6. **Async Worker (C++ userver)** — фоновый обработчик тяжёлых задач (сканирование на вирусы).
7. **Message Broker (RabbitMQ)** — очередь сообщений для асинхронной связи между сервисами (Event-Driven паттерн).
8. **Relational DB (PostgreSQL)** — хранит пользователей, деревья папок и справочник файлов.
9. **S3 Object Storage (MinIO)** — физическое хранилище файлов.

### Архитектурно значимый сценарий: 
Например, асинхронная загрузка файла. При загрузке файла `API Gateway` передаёт поток в `File Node`. Узел синхронно обращается к `User Service` (для проверки наличия свободного места) и к `Directory Service` (для регистрации файла в БД со статусом «На проверке»). Далее файл стримится напрямую в `S3 Storage`. После сохранения `File Node` публикует событие в `RabbitMQ` и отдаёт клиенту успешный ответ (все это время пользователь не блокируется ожиданием). В фоне `Async Worker` забирает задачу, проверяет файл во внешнем антивирусе и обновляет его статус в БД на «Доступен».

### Диаграммы архитектуры

#### System Context (уровень C1)
Диаграмма показывает систему в контексте внешних взаимодействий:
* **Пользователь** взаимодействует с веб-интерфейсом
* **Антивирусное API** — внешний сервис проверки на вирусы

#### Container (уровень C2)
Диаграмма детализирует внутреннюю архитектуру системы:
* **Web UI** — фронтенд на React
* **API Gateway** — маршрутизация запросов (Nginx)
* **User Service** — управление пользователями и квотами (gRPC)
* **Directory Service** — управление иерархией каталогов (gRPC)
* **File Node** — приём и передача файлов
* **Background Worker** — фоновая обработка задач
* **RabbitMQ** — брокер сообщений
* **PostgreSQL** — реляционная база метаданных
* **MinIO** — объектное хранилище файлов

#### Dynamic (AsyncUploadFlow)
Последовательность действий при загрузке файла:
1. Инициирование загрузки пользователем
2. Формирование POST-запроса
3. Маршрутизация потока данных
4. Проверка дисковой квоты
5. Регистрация метаданных в каталоге
6. Транзакция записи в БД
7. Потоковая запись в S3
8. Публикация события
9. Доставка задачи воркеру
10. Антивирусная проверка
11. Обновление статуса файла
